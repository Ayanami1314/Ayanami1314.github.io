压缩和软件设计哲学
2023/10/16

软件设计哲学：

让代码变得复杂的有两个主要因素：

Obscurity and dependency

为了避免Obscurity, 我们需要命名有意义的变量和抽象，比如用枚举代替奇怪的条件常数
为了避免dependency，我们需要避免重复，要做到“对一个bug，只修正一个方法”

管理复杂度：

  1. 编程时要更多地在“战略上”而不是“战术上”思考

  2. 我们无法一次处理完所有复杂度，所以我们需要隐藏复杂度

  3. build things deep: 最好的模块是使用起来简单易懂，但能够执行复杂操作的模块（不要让接口直接和底层绑定，尽可能少让一个模块的内部信息“泄露”到另一个模块）

  4. 警惕“时间分解”，事物发生的顺序并不是一个编程的好顺序，从整体上俯瞰是一个更好的做法

1. 拓扑排序

对于任意的节点i,j 如果存在从i到j的一条边，则j应该排序在i的后面
满足这样性质的排序叫拓扑排序

存在拓扑排序的条件：图是无环的

构建拓扑排序的方法：
对节点引入两个量，入度和出度，代表进入节点和从节点发出的边条数

首先，由于图是无环的，必然可以找到一个入度为0的点（“起点”）
抛去起点和与起点相连的边之后，子图必然有一个新“起点”

不断找这样的“起点”就得到了拓扑排序

算法是O（V+E）的

  1. 计算所有点的入度 O（V+E）
  2. 维护一个队列，包含所有入度为0的点，每次除去一个点和其邻边，并将相邻节点的入度减一（无向图）O（V+E）

python并发
2023/10/16

﻿

并发：多协程、多线程、多进程、多机器

任务：IO密集和cpu密集
IO密集：数据库、文件读写等
cpu密集： 计算密集

多线程：单cpu（前提：python GIL锁），目的是io和cpu分离并行执行，适用于io密集，内存开销比多进程少

多进程：多cpu，目的是多cpu并行计算，适用于cpu密集，内存开销大

线程threading库

t = threading.Thread(target=func, args=(…))
t.start() # 开始线程，执行func
t.join() # 等待线程结束

例如：

threads = []
for _ in range(count):
    thread = threading.Thread(target = f, args=(...))
    threads.append(thread)
for thread in threads:
    thread.start()
for thread in threads:
    thread.join()

如果多个线程之间存在共享资源
需要加锁，
lock = threading.Lock()给需要线程独占的运行函数f套上with lock:

当对一个类的某个方法加锁的时候，需要在__init__上锁self.lock = threading.Lock()
在方法调用之中
with self.lock：
而不是直接在方法调用之中创建lock, 这样无法保证创建lock与线程切换的顺序

生产者-消费者模型：

以爬虫为例：
生产者线程get网络资源，存储到任务序列之中
消费者线程取出任务序列的值，存储到目标文件

任务序列 queue.Queue
例 

buffer = queue.Queue(maxsize = 10)
def producer():
    item = ...;
    buffer.put(item)
def consumer():
    while(True):
        item = buffer.get()
        # do something

线程池：

from concurrent.futures import ThreadPoolExetutor
with concurrent.futures.ThreadPoolExetutor as pool:

    tasks = [...]
    results = pool.map(func, tasks)
    for res in results:
        print(res)
