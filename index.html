CALL（compiler, assembler, linker&loader）
2023/10/16

﻿

  1. diff from Intepreter and compiler
    Interpreter 高级语言–>高级语言
    compiler 高级语言–>机器码/汇编（包含很多的优化）
  2. C flow
    CALL chain
    .c–>compiler–>.s
    .s–>assember(translate pesudo instruction)–>.o(object file)

.o –>linker(.a static lib)–>可执行文件
.exe–>loader(加载进内存)–>输出
(如果使用动态库，link只使用动态库.so解析符号表, 具体的动态库代码、数据不放入.exe的.text 和 .data段之中， 剩下一部分的link放在load的时候执行)

  1. compiler:
    lexer: 词法分析器，把输入
    int foo(int a, double b){
    }
    拆成一系列的token, 比如 “int” “foo” “（”

然后通过构建好的语法树parse tree进行语义分析

assember: 翻译汇编伪指令，将所有指令转换成机器码
机器码之中还附带了一些信息
.header : 一些程序信息，如大小等
.text 代码段
.data 静态和全局变量
debug info 调试信息
relocation table 重定位表，列出了所有assembler不能解决，需要重定位的东西（比如malloc的地址）
symbol table 符号表， 列出了本代码之中的符号

assember 顺带的把j label的label和branch 的 label 变成PC-relative address
这样这些东西就不需要被linker和loader修改

linker

整合代码段、数据段信息
重定位外部函数和静态、全局变量
（遍历整个relocation table,先在symbol table里面找，如果没有，再到外部lib里面找，找到了之后使用绝对的地址替换之）
linker 假定程序均从0x40000000开始，这体现了虚拟内存的概念，load进内存的时候还要改变程序的一些绝对地址

in reality, loader is OS

loader干什么：

  1. 读取linker产生的executable的文件的信息，包括大小等，为它分配一段空间
  2. copy 数据和程序到空间之中
  3. 初始化寄存器，将传入的参数(argc argv)放到栈上
  4. 跳转到开始位置，执行程序，在结束位置通过系统调用结束程序

今日学习：

  1. GAMES101 4，5讲
    第四讲：进一步的变换

  • 三维变换

  • 罗格里德旋转公式，得到任意方向n旋转α角度的公式

  • 进一步的变换：Camera/View Transformation
    mvp（model view projection变换）

    相机和物体的位置关系可能有各种情况，便于研究将相机恒定置于原点，并且其方向为：朝上是Y，正对-Z（这是为了让X取在指向纸面外）

    将相机的两个特征方向（-Z，Y）分别称为
    gaze direction(g) 和 up direction(t)


那么为了保持物体与相机的相对位置不变，物体需要左乘平移矩阵T，再左乘旋转矩阵R

  • 如何求R？
    直接求g，t，gt如何旋转到x,y,z比较困难，但反过来，从x,y,z旋转到g,t,g*t是不难的 ，注意到旋转矩阵的对应元素就是g, t, gt在xyz下的坐标，就得到了R^-1

    更好的是，R作为一个旋转矩阵是正交的Rt = R^-1, 也就是说我们只需要转置！

对projection 又有两种：正交投影（Orthographic）和透视投影（Perspective）

  • 先讲正交投影：正交投影把一个[l,r]*[b,t]*[f,n]（左右下上远近）的物体区域放缩到[-1,1]^3,再取xOy平面

其变换矩阵也很简单，将长方体的中心平移到原点+放缩 M=ST

  • 之后是透视投影，重点在于如何把透视归结到之前的矩阵表述之中去
    一个自然的想法是，我假设有两个面n和f， 其中n是“标准面”，透视的本质是视线交汇于一点而不再平行，如果把f面的xy“放缩”到和n面一样大，那么f这一块nf区域就可以变成正交投影


通过这几个特征推导“压缩”矩阵：
压缩后远平面f上的点的齐次坐标从（x,y,f,1）变成了（x*n/f, y*n/f, f, 1），即M（x,y,f,1)t= （x*n/f, y*n/f, f, 1）t

近平面上不变，M左乘（x’, y’, n, 1）= （x’,y’,n,1）
从这两个式子可以推出M(相差一个常数，注意齐次坐标（tx, ty, tz, t）和（x,y,z,1）是一样的)

需要注意的是这样的投影变换只保证给定近面n和远面f是严格Z不变的，中间的变换出来的结果比实际值偏大

第五讲：光栅化（Rasterization）

我们现在已经完成了mvp变换（model - view - projection）成功得到了一个[-1,1]^3（或者xOy面）的物体投影，如何从这个标准化的投影重建像呢？

先再引入两个常用参数：fovY(Y轴截段的field of view, 垂直可视角度)和aspect ratio（宽高比）

（主要是视锥体里面的参数转换关系，n,t的另一种表述）

Then, 我们要如何将其画在屏幕上？
（屏幕小知识点，分辨率1920*1080就是1080p，etc）

将[-1,1]^2 重新拉伸成 ScreenWidth * ScreenHeight, 之后平移一下，原点重新放回左下角

接下来要做的是离散化，判断一个x,y的点所在的像素是否需要“画”（采样）

以三角形为例，即判断一个点（x,y）是否在三角形内部（通常这个点取每个像素点的中心点，即int x， int y, 然后取x+0.5,y+0.5）

判断点是否在三角形内部的巧妙方法：利用叉积！
AP叉乘AB可以通过正负号判断P在AB左还是右侧，而依次叉乘，点P在三角形三边同侧时，才是内部

之后对内部的点赋值，先取一个包围盒（bounding box），三顶点x最小到最大，y最小到最大（不需要遍历整个屏幕）

2023/10/16

Gitlet设计文档：

.gitlet 不对工作目录进行任何变动，所有的修改都在.gitlet之中进行

.gitlet 应当有以下部分

  • commit：整体的提交树结构，每个树结点应该只储存各个文件的version .getlet/commit
  • (commit) blob：各个文件的每一个修改版本 .gitlet/commit/blobs/file-SHA/version_i, 是序列化的持久对象
  • stage：.getlet/stage
  • track ：一张表，记录已跟踪的文件

提交树有以下部分：

hash 是用来分辨同一个name的不同文件的

文件结构：

CWD
└── .gitlet├── commit <==== 树形结构，每一个commit结点cmNode可遍历 内commit对象，commit
├── blobs ── table <===== 序列化的持久对象的hash table，Map<hash, filepath>，cm的hash值通过这个表查到对应的文件

​ ├── files├── … <====所有的文件

​ ├── stage <===== 缓存区， 存Map<name, file> commit时对同名文件算hash取新
​

代码结构├── main
├── repo
├── blobs

​ ├── stage

第一次重构：

首先，文件被存储在.gitlet/blob之中, 文件名称是commit对象的sha1码, 不需要filename ，之前的是冗余设计；sha1码由文件内容计算得出

blob/commit-sha1/所有的文件(名称是文件的sha1) + 一张<filename， sha1>Map filemap

其次，filemap有此次commit包含所有文件的<sha1, filename> Map, 其他的交给文件结构和cmNode，不保存

保存commit的图结构：

.gitlet/cmMap cmMap是由cmNode组成的一张Graph，

不需要Map数据结构，也不需要考虑序列化的问题，采用文件的方式表达数据结构

.gitlet/cmGraph/cmNode文件夹（内部是各个cmNode文件，文件名为selfID） + 一张表cmMap

每个cmNode只有timeStamp和parentID(commit-sha1)和message三个成员（该次commit的sha1），通过sha1 使用文件名拼接 .gitlet/blob/commit-sha1 可以访问到对应的commit对象;

cmNode（commit）信息：（）

  • timeStamp
  • message
  • parentID (commit-sha1 列表， 除了merge节点都是1个元素)
  • selfID(不是实际的内容，就是文件名)

cmMap类（也是文件名）:

  • branches, branches为Map<branch-name, commit-ID> (不用指针防止序列化和反序列化出问题)
  • HEAD， HEAD为cmNode对象的selfID(sha1), 通过文件名访问对应的cmNode

如果只涉及到单分支信息打印，不需要blobs和cmMap，在cmNode之间解决

如果需要多分支信息打印，cmMap + cmNode, 由cmMap得到分支的commit-ID，之后定位到文件

如果涉及到文件信息更改，再从blob里面，使用commit-sha1直接定位到对应文件夹，使用文件名（通过.filemap 转换成sha1）或者文件的sha1

暂存区：

.gitlet/stage/… 存file

删除区：

.gitlet/removal/… 存file

2.cache
2.1）SRAM, DRAM，闪存和磁盘
速度逐渐减慢，成本逐渐降低
原理：

SRAM：一组直接接外接电源的触发器电路（这意味着功耗和对电源的极高稳定性要求，断电立即丢失数据），因而不需要刷新操作，速度快

DRAM：每个存储单元不直接接外接电源，采用电容的方式存储，但电容会以漏电等方式损失电荷，所以需要定期接电源刷新（只在刷新时接电源，降低功耗；总线接电源而不是每个电容接，降低复杂度）

DRAM之中的存储单元还有bank这种较小的集合，这方便了在DRAM之中的并行操作（不同bank可以同时操作）

闪存:
为了读取和写入数据，闪存芯片中的控制逻辑电路负责管理电荷的注入和释放。当需要读取存储的数据时，控制逻辑电路会检测浮栅中的电荷状态，并将其转换为可读取的信号。当需要写入新的数据时，控制逻辑电路会控制电荷的注入或释放，以改变浮栅的电荷状态。

闪存存储单元中的每个存储单元通常是基于一种称为浮栅（Floating Gate）的结构。浮栅是一个与主要电路隔离的电荷储存区域。当存储单元被要求存储数据时，电荷可以被注入到浮栅中，表示存储的信息。这些电荷在浮栅中被困住并保持不变，即使断开外部电源，电荷也不会自行流失。

但电学存储自带热耗损等，故寿命比磁盘低；但整体结构比较稳定，比磁盘抗摔（）

磁盘：机械结构+磁头，由于有“移动磁头”这样的机械操作，因而速度最慢

2.2）cache的思想:
对任意存储器，存储的容量越大，其访问速度必然越慢（或者说，做到访问速度和小容量的存储器相同，成本变高）

如果CPU直接和内存交换信息，从内存里面读写的典型时间是250个时钟周期，在250个时钟周期内，现代计算机可以执行上千条甚至上万条指令

也就是说，一次 *p = value，相当于上万条 p += 4的执行时间，这是不可忍受的。因而我们需要一种能够更快速的读取内存中数据的方法，cache就诞生了

现代计算机的存储结构是多层的，从上到下是类似SRAM到DRAM到闪存或者磁盘。而所谓cache，即高速缓存技术，就是把常用的数据存放到更小、更快的存储器（称之为缓存cache）之中，CPU读数据的时候先在cache之中找，找到了就不访问内存了，直接取cache之中的值

由于快的前提是小，所以不可能存所有的值；而如果找不到，相当于没有cache，所以关键在于cache里面存什么值、如何更新之，使得使用之中尽可能多的值能够找到

引入理论基础:

  • 时间局部性（temporal locality）：如果某个数据被访问，那么它很有可能会再被访问。e.g. loop
  • 空间局部性（spatial locality）：如果某个数据被访问，临近的数据很有可能会被访问。 e.g. array

因而，考虑时间局部性，被访问的数据可以放进cache（访问到该数据时，更新cache以加入该数据）

考虑空间局部性，在cache之中设计了block的结构。一条cache的数据段未必只能对应一个数据， 可以是一个更大的block, 比如16个单字（对应4条int），也可以结合预取机制（取决于对应软硬件设计），可以做到取a[0]的时候，取下a[0],a[1],a[2],a[3]，再取a[1]的时候就能cache hit

（具体表现看怎么设计的， 比如一个32位的cache在取一个char[] 数组的时候可能会只取一个char, 其他的补0， 也可能会取多个char,看设计）

需要注意的是, block也不是越大越好，block（一条cache）是数据在存储层次之中传输的基本单位，更大的block必然会使数据在存储层次间传输消耗更多的时间

2.3）实际cache示例
2.3.1 直接映射(direct mapped) cache:
怎么存cache呢？
最简单的方法是就按内存的地址，有一条加一条

如：
0xabcdef00 …(第一条cache)
0x0012546 …(第二条cache)
…

但这样存，如果要找某一条数据，就必须逐条比对，很是铸币

联想到现实之中的图书馆，我们需要对每一条cache重新编号，并且按编号进行二分之类查找，这样找的才会快

并且，由于数据本身就是2进制的，“2分查找”的实现都可以直接交给硬件，十分简单
上面不是最主要的原因，为cache重新编号的目的是使不同地址上的数据尽可能均匀分布（映射）到cache之中，以尽可能提高cache利用率和命中率

（这也是为什么index要选地址的小位而不是大位的原因）

一种做法是，假设我的cache 最大条数是8，那么我的地址mod 8, 就是我的cache编号index
（这样的cache最大容量很自然应该是二的幂，对地址取模只需要取末几位即可）
0xfffffff 这条地址映射到cache编号为 111 （f: 1111）
0x12345678 这条地址映射到cache编号为000 (8: 1000)
…

这样会把多条地址上的值映射到一个cache上，所以我们需要比较地址的其他值来确定要求的数据是否就是cache之中的数据。对应的，一条cache之中有一个lag标签，存了0xffffffff除了111（index）之外的其他部分。比对lag即可判别是否相同。

还有一些细节，比如：程序开始的时候，cache里面的值并没有清空，这些乱七八糟的值应该有一个“是否有效”的标识符valid，占一位

又比如：内存之中一定要按字节对齐，并且很多时候更进一步，按字对齐（4Byte，此时一个单独的char占了1Byte 和3个空Byte），这可以不存在lag之中；如果block不止一个字，比如4个字，那么地址之中还需要分离2位出来，指示某个特定的字在block之中的位置

示例：64位存储地址，直接映射cache，cache最大容纳数据大小16KiB，cache一个block 4字，cache所占的实际空间是多少？

解析：
16KiB/4字 = 1024 条cache = 2^10
因而64位地址末10位是cache index
64位存储地址 -10- 2（内存对齐）-2(指示一个block之中单字位置,2*2=4) = 50位 lag

[50 + 1(valid位) + 4*32（block）]*1024 =179 Kib = 179/8 KiB = 22.4KiB

2.3.2 N-way set-associative cache
上面的直接映射cache也有铸币的一点

如果来回访问不同地址，但是相同index的内存块，就会全miss, 0 hit

例如
4 lines cache， 数据段32位(意味着 地址扣去对齐后末两位是index)

如果有这样的long long int array (4 word / unit)
0 4 0 4 0 4 0 4…

起始位置不妨设为0x12345678

1：存0， index = 00（8：1000）
2：存4， index = 00 (8+4: 1100), cache miss, 用1 替换 00 line的cache存的0
3: 用0替换4……
etc, 称为ping-pong effect， all miss

更常见的实例代码如下

int x[12]={...};  // 12可换成任意4的倍数
int y[12]={...};
for(int i=0;i<12; i++){
    z[i] = x[i]*y[i];
}
// 这个程序看起来很有空间相关性，理应得到很高的cache hit rate，但是，如果cache是direct-mapped cache,以4个block为例，由于x，y都是放在堆上且相邻定义，xy地址差的是32*4位的整数倍，两个数组的index相同，存入cache时会存入x，然后y cache miss,存入y，然后x cache miss，......

为了弥补这种铸币的错误，出现了N way set-associative cache
其思想为，将1个index对应1个cache block变为1个index 对应N个cache block

因而我们需要N个比较器，现在存0，放到index 00 block 1之中，存4， 放到index 00 block 2之中，…，能够解决周期<N+1的ping-pong 问题
读的时候，对N个block都比较一遍lag
由于比较器很贵，所以不是N越大越好
现代处理器N一般为8， 16，32等

2.4 写cache
先前的都是读，写主要有两种
write-through 和 write-back

write-through: 每次写都同步更新cache和实际内存
优点：数据可恢复（即使断电，内存里面也有）
缺点：多次中断流水线，大大降低性能

write-back：每次写，分为多种情况
核心思想是，将不必要立即改变的更新都先“存”到缓冲区，缓冲区满或者cache的该set满了，又或者要读一个已经被修改过的cache位时，才将缓冲区的数据写到内存，尽可能减少对流水线的打断

优点：快
缺点：硬件实现复杂，成本高

(write-back变种：如果读写共用一套cache, 那么前面说的读时就不需要从cache写回内存 ，那就是unified cache了，这样的好处是写性能是提高了，但也出现了很多新问题，比如优化策略、cache存储管理等)

引入一个dirty bit来表示这个数据有没有被写过，如果一个数据是dirty的，读到它时先将写缓冲区的数据写回内存，再读内存、存入读cache（也就是说读之前需要在写那边检测一遍dirty）

写时，写缓冲区满或者该cache block将要被替换之前一直写就行（还有设置dirty），如果写缓冲区满了或者该cache block将要被替换，写回内存；

实际上的CPU基本都是write-back的

几种cache (set)满了替换cache的策略：

  • LRU (Least Recently Used) — When we decide to evict a cache block to make space, we select the block that has been used the furthest back in time of all the blocks.
  • Random - When we decide to evict a cache block to make space, we randomly select one of the blocks in the cache to evict.
  • MRU (Most Recently Used) — When we decide to evict a cache block to make space, we select the block that has been used the most recently of all the blocks.

2.4 对实际code的启示
1.如果要高性能，少写list和tree, 指针式的寻址模式不符合cache，即使同样都是O（N），遍历一个链表比遍历一个数组慢几十上百倍也很正常（事实上，一般标准库的给的接口都是array式的tree, map, list， 如ArrayList而不是LinkList,又比如做hash的时候，各个桶实际上是数组方式排列的）。尽量多使用连续的内存，连续的读写

2.能用临时变量保存中间结果，不要开数组之类

3.数组尽量做步长为1的遍历，少做函数调用（sp指针读写）——为什么快排比堆排、归并快：和归并比，更好利用cache（归并需要新数组，2倍空间相当于减半cache），并且多次函数调用；和堆排比，大长度情况下堆排几乎没有cache

4.矩阵相乘的重新排序：列寻址的数组没有利用cache，同时，使用临时变量而不是一直改变数组的值会明显快

// 最一般的实现
for(int i=0;i<n;i++){
    for(int j=0;j<n;j++){
        int sum = 0;  // 用临时变量sum避免不断读写C[i][j], 提升性能
        for(int k=0;k<n;k++){
            sum += A[i][k] * B[k][j];// 这里不好，B的列寻址不符合二维数组在内存之中的存储法，n大时，会全是cache miss
        }
        C[i][j] = sum;
    }
}


// 改进的实现
for(int i=0;i<n;i++){
    for(int k=0;k<n;j++){
        for(int j=0;j<n;k++){
            C[i][j] += A[i][k] * B[k][j]; //利用了cache
        }
    }
}
// 这里利用了Cache 但是增加了读写C[i][j]的时间开销，事实证明这是完全tradeoff的，并且，由于预存的硬件机制，C[i][j]的读写实际上也能cache

5.条件传送与条件控制：尽可能使用条件传送，尤其是当条件实质上不可预测时

// 如果a[i]<b[i]，a[i]=min, b[i]= max，一个性能上不好的实现
if(a[i]<b[i]){
   // do nothing
}
else{
    int t = a[i];
    a[i] = b[i];
    b[i] = t;
}

//良好的实现 
int min = a[i]<b[i] ? a[i]:b[i];
int max = a[i]<b[i] ? b[i]:a[i];
a[i] = min;
b[i] = max;

// 原因：回想流水线的知识，当执行分支跳转的时候，如果分支预测错误，有严厉的惩罚时间。

本质上， 第一种写法相当于a[i]<b[i]的结果，进入一个multiplexer，multiplexer的两个输入是PC+4和PC+offset，决定的是PC跳转。而a[i]和b[i]的大小很多情况下不可预测。

第二种写法相当于进入multiplexer 的不是PC+4 和 PC+ offset， 而是a[i]和b[i]，没有跳转，自然没有罚时

6.能用临时变量保存的结果，不要有额外的开销
例：

 for(int i=0;i<strlen(s);++i) {
 //很烂的写法，每次都会算一遍strlen(s),如果s长不可接受

而假设f(s)改变条件的同时也更新j,编译器如果把它移到函数外面，那么j的值就会出问题。
