CALL（compiler, assembler, linker&loader）
2023/10/16

﻿

  1. diff from Intepreter and compiler
    Interpreter 高级语言–>高级语言
    compiler 高级语言–>机器码/汇编（包含很多的优化）
  2. C flow
    CALL chain
    .c–>compiler–>.s
    .s–>assember(translate pesudo instruction)–>.o(object file)

.o –>linker(.a static lib)–>可执行文件
.exe–>loader(加载进内存)–>输出
(如果使用动态库，link只使用动态库.so解析符号表, 具体的动态库代码、数据不放入.exe的.text 和 .data段之中， 剩下一部分的link放在load的时候执行)

  1. compiler:
    lexer: 词法分析器，把输入
    int foo(int a, double b){
    }
    拆成一系列的token, 比如 “int” “foo” “（”

然后通过构建好的语法树parse tree进行语义分析

assember: 翻译汇编伪指令，将所有指令转换成机器码
机器码之中还附带了一些信息
.header : 一些程序信息，如大小等
.text 代码段
.data 静态和全局变量
debug info 调试信息
relocation table 重定位表，列出了所有assembler不能解决，需要重定位的东西（比如malloc的地址）
symbol table 符号表， 列出了本代码之中的符号

assember 顺带的把j label的label和branch 的 label 变成PC-relative address
这样这些东西就不需要被linker和loader修改

linker

整合代码段、数据段信息
重定位外部函数和静态、全局变量
（遍历整个relocation table,先在symbol table里面找，如果没有，再到外部lib里面找，找到了之后使用绝对的地址替换之）
linker 假定程序均从0x40000000开始，这体现了虚拟内存的概念，load进内存的时候还要改变程序的一些绝对地址

in reality, loader is OS

loader干什么：

  1. 读取linker产生的executable的文件的信息，包括大小等，为它分配一段空间
  2. copy 数据和程序到空间之中
  3. 初始化寄存器，将传入的参数(argc argv)放到栈上
  4. 跳转到开始位置，执行程序，在结束位置通过系统调用结束程序

GAMES101四五讲
2023/10/16

今日学习：

  1. GAMES101 4，5讲
    第四讲：进一步的变换

  • 三维变换

  • 罗格里德旋转公式，得到任意方向n旋转α角度的公式

  • 进一步的变换：Camera/View Transformation
    mvp（model view projection变换）

    相机和物体的位置关系可能有各种情况，便于研究将相机恒定置于原点，并且其方向为：朝上是Y，正对-Z（这是为了让X取在指向纸面外）

    将相机的两个特征方向（-Z，Y）分别称为
    gaze direction(g) 和 up direction(t)


那么为了保持物体与相机的相对位置不变，物体需要左乘平移矩阵T，再左乘旋转矩阵R

  • 如何求R？
    直接求g，t，gt如何旋转到x,y,z比较困难，但反过来，从x,y,z旋转到g,t,g*t是不难的 ，注意到旋转矩阵的对应元素就是g, t, gt在xyz下的坐标，就得到了R^-1

    更好的是，R作为一个旋转矩阵是正交的Rt = R^-1, 也就是说我们只需要转置！

对projection 又有两种：正交投影（Orthographic）和透视投影（Perspective）

  • 先讲正交投影：正交投影把一个[l,r]*[b,t]*[f,n]（左右下上远近）的物体区域放缩到[-1,1]^3,再取xOy平面

其变换矩阵也很简单，将长方体的中心平移到原点+放缩 M=ST

  • 之后是透视投影，重点在于如何把透视归结到之前的矩阵表述之中去
    一个自然的想法是，我假设有两个面n和f， 其中n是“标准面”，透视的本质是视线交汇于一点而不再平行，如果把f面的xy“放缩”到和n面一样大，那么f这一块nf区域就可以变成正交投影


通过这几个特征推导“压缩”矩阵：
压缩后远平面f上的点的齐次坐标从（x,y,f,1）变成了（x*n/f, y*n/f, f, 1），即M（x,y,f,1)t= （x*n/f, y*n/f, f, 1）t

近平面上不变，M左乘（x’, y’, n, 1）= （x’,y’,n,1）
从这两个式子可以推出M(相差一个常数，注意齐次坐标（tx, ty, tz, t）和（x,y,z,1）是一样的)

需要注意的是这样的投影变换只保证给定近面n和远面f是严格Z不变的，中间的变换出来的结果比实际值偏大

第五讲：光栅化（Rasterization）

我们现在已经完成了mvp变换（model - view - projection）成功得到了一个[-1,1]^3（或者xOy面）的物体投影，如何从这个标准化的投影重建像呢？

先再引入两个常用参数：fovY(Y轴截段的field of view, 垂直可视角度)和aspect ratio（宽高比）

（主要是视锥体里面的参数转换关系，n,t的另一种表述）

Then, 我们要如何将其画在屏幕上？
（屏幕小知识点，分辨率1920*1080就是1080p，etc）

将[-1,1]^2 重新拉伸成 ScreenWidth * ScreenHeight, 之后平移一下，原点重新放回左下角

接下来要做的是离散化，判断一个x,y的点所在的像素是否需要“画”（采样）

以三角形为例，即判断一个点（x,y）是否在三角形内部（通常这个点取每个像素点的中心点，即int x， int y, 然后取x+0.5,y+0.5）

判断点是否在三角形内部的巧妙方法：利用叉积！
AP叉乘AB可以通过正负号判断P在AB左还是右侧，而依次叉乘，点P在三角形三边同侧时，才是内部

之后对内部的点赋值，先取一个包围盒（bounding box），三顶点x最小到最大，y最小到最大（不需要遍历整个屏幕）

Gitlet设计文档v2
2023/10/16

Gitlet设计文档：

.gitlet 不对工作目录进行任何变动，所有的修改都在.gitlet之中进行

.gitlet 应当有以下部分

  • commit：整体的提交树结构，每个树结点应该只储存各个文件的version .getlet/commit
  • (commit) blob：各个文件的每一个修改版本 .gitlet/commit/blobs/file-SHA/version_i, 是序列化的持久对象
  • stage：.getlet/stage
  • track ：一张表，记录已跟踪的文件

提交树有以下部分：

hash 是用来分辨同一个name的不同文件的

文件结构：

CWD
└── .gitlet├── commit <==== 树形结构，每一个commit结点cmNode可遍历 内commit对象，commit
├── blobs ── table <===== 序列化的持久对象的hash table，Map<hash, filepath>，cm的hash值通过这个表查到对应的文件

​ ├── files├── … <====所有的文件

​ ├── stage <===== 缓存区， 存Map<name, file> commit时对同名文件算hash取新
​

代码结构├── main
├── repo
├── blobs

​ ├── stage

​ ├── commit

​ └── dump

第一次重构：

首先，文件被存储在.gitlet/blob之中, 文件名称是commit对象的sha1码, 不需要filename ，之前的是冗余设计；sha1码由文件内容计算得出

blob/commit-sha1/所有的文件(名称是文件的sha1) + 一张<filename， sha1>Map filemap

其次，filemap有此次commit包含所有文件的<sha1, filename> Map, 其他的交给文件结构和cmNode，不保存

保存commit的图结构：

.gitlet/cmMap cmMap是由cmNode组成的一张Graph，

不需要Map数据结构，也不需要考虑序列化的问题，采用文件的方式表达数据结构

.gitlet/cmGraph/cmNode文件夹（内部是各个cmNode文件，文件名为selfID） + 一张表cmMap

每个cmNode只有timeStamp和parentID(commit-sha1)和message三个成员（该次commit的sha1），通过sha1 使用文件名拼接 .gitlet/blob/commit-sha1 可以访问到对应的commit对象;

cmNode（commit）信息：（）

  • timeStamp
  • message
  • parentID (commit-sha1 列表， 除了merge节点都是1个元素)
  • selfID(不是实际的内容，就是文件名)

cmMap类（也是文件名）:

  • branches, branches为Map<branch-name, commit-ID> (不用指针防止序列化和反序列化出问题)
  • HEAD， HEAD为cmNode对象的selfID(sha1), 通过文件名访问对应的cmNode

如果只涉及到单分支信息打印，不需要blobs和cmMap，在cmNode之间解决

如果需要多分支信息打印，cmMap + cmNode, 由cmMap得到分支的commit-ID，之后定位到文件

如果涉及到文件信息更改，再从blob里面，使用commit-sha1直接定位到对应文件夹，使用文件名（通过.filemap 转换成sha1）或者文件的sha1

暂存区：

.gitlet/stage/… 存file

删除区：

.gitlet/removal/… 存file

对于所有在.gitlet/stage之中的文件：计算sha1值(file-sha1)，如果不在副本之中，在.gitlet/blob/commit-ID 之中保留文件的副本，名称为”f_”+file-sha1

构建commit对象c，parentID是HEAD

C的内存管理机制：
一、概览
对多程序同时并行：虚拟内存
对其中一个程序：
stack, heap

function + local var –> stack
last in, first out
函数不断调用
funcA
var in funA
func B
var in funB
funC
…
当函数return的时候回收stack 空间
（因而返回一个local var的指针是错误的，相当于在stack上占了一个位置，这个位置此时也没有东西了；free一个stack上的变量也是错误的）

heap：三个存储区对应三种数据：

  1. 自由存储区，对应malloc
  2. 静态存储区，存global + static
  3. 常量存储区，不允许修改，存常量

二、malloc、calloc、realloc

直接与操作系统对接，让操作系统在堆区域分配一块给定大小的区域

malloc:分配，内部是随机值
calloc:分配之后初始化为0
realloc:重量级。如果此时指向的内存有足够的空间，则就是单纯的加长；如果空间不够，会重新找一个够的区域，相当于malloc新区域，copy原区域，free 原始区域内存

但假设有void* p1 = malloc(sizeof(int)2)
void p2 = p1;
p1 = realloc(…)
p2不会随着改变！如果调用p2，会让程序崩溃

为什么free只有一个参数？free如何知道是指向什么的指针？（所有指针都是本质上void*，单纯一个地址值）

答：这就是free一定要搭配malloc的原因，malloc之后存储了一些额外的标记，free能捕获之

三、内存泄漏
不只是空间问题，程序结束时会释放所有占用的空间，对于小型程序，空间不是问题

但内存泄漏会带来以下的问题：

  1. 堆之中，大块的连续内存减少，使得程序速度减慢
  2. 代码安全性–缓存区攻击等
    （例如在一个程序中，有一个缓冲区用于存储用户输入的字符串，程序会将该字符串复制到一个指定的内存地址中。如果攻击者输入的字符串长度超过了程序预期的缓冲区长度，那么程序可能会将超出缓冲区长度的数据写入相邻的内存区域中，这有可能包括程序的代码区域。攻击者可以构造一段恶意代码，将其插入到超出缓冲区长度的内存区域中，并通过修改指向该代码的指针，使程序执行这段恶意代码。修改指针可以将程序的控制流从正常的代码路径转移到恶意代码路径，从而实现攻击者的目标。攻击者可以利用缓存区溢出漏洞或其他漏洞，溢出缓存区并覆盖代码指针，从而将程序的控制流重定向到恶意代码路径。攻击者可以通过这种方式执行任意代码，例如窃取敏感信息、篡改数据、破坏系统稳定性等。）

浮点数计算
IEEE754标准
第一位0 + 1 -
后面float32 –> 8位指数， 23位小数
double64 –> 11位指数，52位小数
-1^s * (1 + F)^exponent

精度和range不能得兼，类似科学计数法，在表达极大或极小的数的时候，重要的只是几位有效数字

牺牲精度换范围

二进制 “科学计数” 都是 1.xxx*2^k
因而可以忽略1.，只表示小数部分xxx(F)

有效数字F：无符号正常编码
指数exponent: 无符号正常编码
-127-127 –> 0-256
不用补码的原因是这样比较方便比较

这样设计比较两个浮点数不需要考虑浮点部分，比较指数从高到低位即可

表达特殊数的设计：
正负inf: 浮点数0， 指数位全1

NaN：浮点数不为0，指数全1
规定NaN的>< == 比较全为false
(特例NaN == NaN)是未定义行为

0：浮点数为0，指数位全0
对于介于最小表示2^-127和0之间的数，不采用自动补1，即允许不是科学计数，直接使用F而不是1+F，这样补全了中间的空间

0.0000001*2^127
