linker

2023/10/16 入门 体系结构

简述：linker入门(除动态链接部分)：

建议直接看CSAPP 第七章，别看我这坨答辩

（一）为什么需要linker:

你也不想改了一个源文件之后需要重新编译整个标准库吧，你也不想改了一个注释之后需要重新编译整个浏览器内核吧（doge）

linker使分立编译成为可能

（二）linker在干啥：

省流：
分立的.o(.s产生，可重定位目标文件，gcc -c main.c产生的文件)文件+外部库（动态库、静态库）

被链接器全部链接起来，顺带帮你把里面大部分地址（除了动态库）全部重定向到内存，让你把程序放进内存就能跑

（三）linker咋干的：

程序被编译器编译成assembly之后，留下了一些空位（重定向relocation），形成的是可重定向文件

linker的主要任务：
1.符号解析
2.重定位

符号解析什么？
外部的函数和数据——a.c 要引用b.c的全局变量，extern之类，现在单独变成了a.o和b.o，这些地方还是空的

又比如printf，程序如何知道printf是啥，从哪个库来，进而找到printf在哪里？在汇编代码之中，现在调用这个函数的地方暂时只有一个抽象的符号“printf”

把printf这个”符号”和实际函数定义对应起来的就是符号解析

重定位：编译器和汇编器不知道电脑上哪些内存是可用的，产生的都是清一色的从某个一样的地址开始的汇编代码。
一是需要重新选地方（和操作系统结合）
二是现在符号已经对应到一个定义了，要在代码和数据之中将它们出现的地方用实际的内存操作替换

(四) 一些应用的重要细节

linker解析符号的过程依照如下的规则：

  1. 强弱符号：有定义的是强符号，没定义（只有声明）的是弱符号。如：int func(int x){return x;} 的func是强符号，int func;的func是弱符号，int func=0；的func是强符号
    （注意，局部变量是存在栈上的，只有全局和static数据+函数有符号）

  2. 多个强符号————错误
    一个强符号————解释成强符号
    如a.c 之中有(global) int x；b.c之中 有（global）double x=1.1; 在link a.o b.o的过程之中，x都会被解释成1.1
    没强符号——随机解释成一个弱符号

  3. 函数重载：函数符号又有细微差别，编译器（compiler）在函数重载的过程中实际上会改变函数名称
    如

class Foo{
     int func(int a, double b);
     int func(string s);
}

编译之后func会变成 func__3Fooid
和func__3Foos
其中，3是Foo的字母个数， i d s是int， double, string的缩写

  1. （静态）库依赖的细节
    假设有静态库lib1.a和目标文件main，且main依赖于lib1.a，链接的时候，一定是从左到右先main

原因：链接器链接的时候是这样的过程：
先建立三个符号集：可重定位目标文件集合E，未解析符号集合U，已定义符号集合D
其中，E上所有内容会被整合成可执行文件

三个集合最开始均为空！

对命令行上输入文件f（从左到右），
如果f是目标文件，则将f添加到E，根据f修改U、D
如果f是库文件，对U之中的所有元素，查找f之中有没有在库之中成员m与之匹配
如果有，那么将m加入E，并更改U、D

如果到最后U不空，报错

如果先加lib1.a, 此时不会加入E！（这是为了只在可执行文件之中包含用到的库文件而不是整个库（太大了））E、U、D之中，则加入main之后，U不空，编译报错

进一步
假设库A.a之中的class A中funcA引用了库B.a中class B的funcShare,class B中引用了class A中的funcShare2

我现在有目标文件C.o， 依赖于class A, 如何链接？
gcc C.o A.a B.a A.a -o C
第一次加A.a的时候，A.a本身还有未定义符号（funcShare），所以要在加了B.a之后，再加一次A.a

(五) 格式与符号基础知识

可执行文件：ELF(linux&Unix), PE(Windows)
静态库: .a(archive,存档)
动态库: .so, .dll

工具和指令：

  1. ar 创建静态库
    linux> ar rcs lib.a obj1.o obj2.o

  2. 构建动态库（共享库，减少静态库更新和每个.exe都要带一个占空间的麻烦）
    linux> gcc -shared -fpic -o sharedlib.so obj1.o obj2.o

  3. 读ELF文件
    linux> readelf -h(文件头) 或-l(程序头) -S（节头）-s(符号表) 等等

  4. 反汇编器
    linux>objdump -d obj.o
