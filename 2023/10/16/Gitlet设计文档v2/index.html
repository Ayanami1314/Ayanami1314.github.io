Gitlet设计文档：

.gitlet 不对工作目录进行任何变动，所有的修改都在.gitlet之中进行

.gitlet 应当有以下部分

  • commit：整体的提交树结构，每个树结点应该只储存各个文件的version .getlet/commit
  • (commit) blob：各个文件的每一个修改版本 .gitlet/commit/blobs/file-SHA/version_i, 是序列化的持久对象
  • stage：.getlet/stage
  • track ：一张表，记录已跟踪的文件

提交树有以下部分：

hash 是用来分辨同一个name的不同文件的

文件结构：

CWD
└── .gitlet├── commit <==== 树形结构，每一个commit结点cmNode可遍历 内commit对象，commit

​ ├── blobs ── table <===== 序列化的持久对象的hash table，Map<hash, filepath>，cm的hash值通过这个表查到对应的文件

​ ├── files├── … <====所有的文件

​ ├── stage <===== 缓存区， 存Map<name, file> commit时对同名文件算hash取新

​

代码结构├── main
├── repo
├── blobs

​ ├── stage

​ ├── commit

​ └── dump

第一次重构：

首先，文件被存储在.gitlet/blob之中, 文件名称是commit对象的sha1码, 不需要filename ，之前的是冗余设计；sha1码由文件内容计算得出

blob/commit-sha1/所有的文件(名称是文件的sha1) + 一张<filename， sha1>Map filemap

其次，filemap有此次commit包含所有文件的<sha1, filename> Map, 其他的交给文件结构和cmNode，不保存

保存commit的图结构：

.gitlet/cmMap cmMap是由cmNode组成的一张Graph，

不需要Map数据结构，也不需要考虑序列化的问题，采用文件的方式表达数据结构

.gitlet/cmGraph/cmNode文件夹（内部是各个cmNode文件，文件名为selfID） + 一张表cmMap

每个cmNode只有timeStamp和parentID(commit-sha1)和message三个成员（该次commit的sha1），通过sha1 使用文件名拼接 .gitlet/blob/commit-sha1 可以访问到对应的commit对象;

cmNode（commit）信息：（）

  • timeStamp
  • message
  • parentID (commit-sha1 列表， 除了merge节点都是1个元素)
  • selfID(不是实际的内容，就是文件名)

cmMap类（也是文件名）:

  • branches, branches为Map<branch-name, commit-ID> (不用指针防止序列化和反序列化出问题)

  • HEAD， HEAD为cmNode对象的selfID(sha1), 通过文件名访问对应的cmNode
    如果只涉及到单分支信息打印，不需要blobs和cmMap，在cmNode之间解决

如果需要多分支信息打印，cmMap + cmNode, 由cmMap得到分支的commit-ID，之后定位到文件

如果涉及到文件信息更改，再从blob里面，使用commit-sha1直接定位到对应文件夹，使用文件名（通过.filemap 转换成sha1）或者文件的sha1

暂存区：

.gitlet/stage/… 存file

删除区：

.gitlet/removal/… 存file

细节实现：

追踪：用文件名称

init:

创建各个文件夹和文件(cmGraph, cmNode, cmMap, .gitlet, .blobs, stage, removal,.filemap)，创建初始提交

add:

检查CWD

若存在，则复制该文件到.gitlet/stage

commit:

通过parentID 找到blob里面的parent commit文件 blob/parentID/*
复制一个包含的文件的sha1列表 的副本

从时间戳+message，计算本次commit的sha1值(ID)， 创建本次commit文件夹 blob/commit-ID

对于所有在.gitlet/stage之中的文件：计算sha1值(file-sha1)，如果不在副本之中，在.gitlet/blob/commit-ID 之中保留文件的副本，名称为”f_”+file-sha1

构建commit对象c，parentID是HEAD

更新HEAD为ID

更新.gitlet/cmGraph

清空删除区

status:

从cmMap找到parent commit

复制一个包含的文件 的副本 <filename, sha1> Map m;

对于.gitlet/stage之中的所有文件：看filename在不在副本m里面，如果不在m之中，那么就是新加已跟踪；如果在，比较sha1, 如果改变就是更改已跟踪(modified)；否则就是未更改

再扫一遍m.keySet(), 如果在CWD的文件夹下不存在同名文件，则是删除（deleted）Modification not staged

建立一个CWD的文件名Set（除了.gitlet）

遍历parentID下的文件列表和stage内的文件列表，移除CWD的同名文件

若CWDSet非空，即为（新加）未跟踪文件

log:

当前节点HEAD，直接在.gitlet/cmGraph/cmNode里面使用parentID名字遍历(如果有merge节点取其中1个)

反序列化读取，打印信息

global-log:

对.gitley/cmGraph/cmNode/* 反序列化读取，打印信息

branch:

cmMap 之中添加branch即可

rm :

如果又没被跟踪，又没暂存就没理由删除

如果该文件没被commit，删暂存区， 不删工作目录

否则，在暂存区内加上该文件，删除存储的文件和工作目录文件

先看上一次commit有没有追踪，用name

其次，判断文件是否在stage区域，用name

find + message:

遍历.gitlet/cmGraph/cmNode/*, 反序列化比对message, 输出所有符合的

checkout：

  1. checkout + –filename 取当前节点的file替换cwd ，计算完file-sha1之后直接blob/HEAD/file-sha1定位

  2. checkout + commit-id + – +filename 取commit节点的file，如果commit-id是16位，直接blob/commit-id/filename取，如果是简略版，需要在blob文件夹里面取名字（commit-id）扫一遍

  3. checkout +branch-name 从cmMap之中的branch（Set）里面取， 之后branch-name->commit-sha1,之后同1，改变branch,重新写回

当前未跟踪且会被覆盖在单条路是不可能的，但在多分支情况，A分支还未提交的文件B分支未跟踪正常

rm-branch：

判断是否是当前分支，

改变.gitlet/cmGraph/cmMap 里面的branches成员，写回

reset:

调用status判断是否存在（新加）未跟踪的文件

若不存在，可以reset, 遍历CWD reset filename

清除暂存区

merge:

合并当前节点A和节点B

快速合并：一条线上，移动HEAD到给定commit处即可（重新赋值HEAD）

多路合并：

先找最佳共同祖先C：以A为起点，对整张图做一个bfs，标记各个节点的距离；再以b为起点，标记各个节点的距离。选出两者距离和最小的点；最佳祖先可能有多个点，如果有多个点，递归地合并这多个点，选一个合并感觉就行

扫一遍split节点，取对应的other、HEAD文件

扫一遍other

main函数：

不实现具体的操作，处理命令符，调用repo的函数

repo:

  • 处理输入，根据输入调用相应类的函数
  • 计算SHA-1值和时间戳
  • 序列化和反序列化

help :

各个命令：

Gitlet init:

创建初始提交和.getlet文件夹，之后再次调用检测.gitlet文件夹是否存在

创建master

Gitlet add + name1, name2, …(*args):

  1. 找name对应的文件
  2. 计算help
  3. 序列化，存到缓存区，更新追踪

Gitlet commit:

commit类有一个TreeMap<name, hash>

blobs 文件名是 hash， 文件内容是file, find的时候直接由 hash找到对应的file (HashSet) 复制上一次commit的文件
将stage内的非重复文件copy到commit区

  1. 存入日志，更新提交树

  2. 清空缓存区

Gitlet log:

  1. 由head指针开始，遍历整个commit tree打印

Gitlet rm + 文件名

移除该文件的跟踪和暂存（如果有的话）

Gitlet global-log

单纯显示所有提交，不要求顺序，遍历整个提交树即可

Gitlet find

类似global-log 遍历

Gitlet status

读取当前分支（提交树的方法）

打印所有分支，打印缓存区

遍历CWD内所有文件（除自己的.gitlet），按字符排序

和当前commit结点比较

多了的，三种情况：如果stage, 如果未缓存但已经跟踪（和上一次commit节点不一样），如果未跟踪

再用跟踪表格反向扫一遍，得到删改了的文件，如果缓存了，就是removed, 如果未缓存，就是 Modifications Not Staged For Commit

Gitlet checkout

不使用遍历，调用一个存好的由<key=SHA-1 ，val = pointer >的Hashmap直接得到目标的commit结点

签出文件：由该结点的文件路径（指针？序号？），得到结点的文件

签出整个结点： 同上

Gitlet branch + name

新建一个指向当前结点的指针

Gitlet rm-branch

？？ 什么意思，只删指针

Gitlet reset commit-id

首先检查所有文件是否已经被跟踪

删除所有文件，checkout 目标commit, 移动头部指针
