大端：相反
内存对齐：
内存对齐（Memory Alignment）是计算机系统中一种常用的数据存储方式，它指的是将数据存储在内存中时，按照某个特定的规则进行排列，以便于处理器访问和处理数据。具体来说，内存对齐是指将数据存储在内存中时，将其放置到内存地址是其大小的整数倍的位置上。

进行内存对齐的主要原因是处理器在访问内存时通常是按照一定的块大小进行的，而内存对齐可以确保数据存储在块的开始位置上，从而减少处理器访问内存时需要读取的字节数，提高内存读写的效率。例如，在32位处理器中，通常采用4字节对齐的方式，即数据存储在内存地址是4的倍数的位置上。

﻿

1. 拓扑排序

对于任意的节点i,j 如果存在从i到j的一条边，则j应该排序在i的后面
满足这样性质的排序叫拓扑排序

存在拓扑排序的条件：图是无环的

构建拓扑排序的方法：
对节点引入两个量，入度和出度，代表进入节点和从节点发出的边条数

首先，由于图是无环的，必然可以找到一个入度为0的点（“起点”）
抛去起点和与起点相连的边之后，子图必然有一个新“起点”

不断找这样的“起点”就得到了拓扑排序

算法是O（V+E）的

  1. 计算所有点的入度 O（V+E）
  2. 维护一个队列，包含所有入度为0的点，每次除去一个点和其邻边，并将相邻节点的入度减一（无向图）O（V+E）
    故总共是O（V+E）

2. 图API与设计

图：一般算法里面说的图都是简单图，即两个节点之间只有一条边；图可以有很多分法，有环图/无环图，有权图/无权图等等

图一般表示成以下的元素：节点vertex和边edge（边一般用（u,v）表示）

首先是图的生成和添加：

public class Graph {
  public Graph(int V):               // Create empty graph with v vertices
  public void addEdge(int v, int w): // add an edge v-w
  Iterable<Integer> adj(int v):      // vertices adjacent to v
  int V():                           // number of vertices
  int E():                           // number of edges
...

对一个图可以有很多的问题，从而有更多的算法：

  • s-t Path: Is there a path between vertices s and t?
  • Connectivity: Is the graph connected, i.e. is there a path between all vertices?
  • Biconnectivity: Is there a vertex whose removal disconnects the graph?
  • Shortest s-t Path: What is the shortest path between vertices s and t?
  • Cycle Detection: Does the graph contain any cycles?
  • Euler Tour: Is there a cycle that uses every edge exactly once?
  • Hamilton Tour: Is there a cycle that uses every vertex exactly once?
  • Planarity: Can you draw the graph on paper with no crossing edges?
  • Isomorphism: Are two graphs isomorphic (the same graph in disguise)?
  • ……

但是我们不希望把图的API搞得如此复杂，所以一个好的图的API不应该包含具体的算法实现，而把实现的东西变成一个接收图的函数之类

例如，在下面的BFS或者DFS（广度/深度优先搜索），我们可以把它写成一个接受两个参数：Graph g, vertex source 的一个function； 在搜索之中产生的路径Path我们也应该写成包含这个函数方法的类内部的一个成员, 而不是把路径放到图API里面去

那我们应该在图的API里面放什么呢？
比如：

  • .adj（vertex v）方法，返回一个可迭代的列表或者集合之类，表示和某个结点相连的点
  • 一个通过某种数据结构（通常是一个最小堆）实现的边的排好序的集合（这对最小生成树很有用）
  • isConnected（vertex u, vertex v）判断两个节点是否相连（内部并查集）
  • 计算某个节点的入度和出度
  • …….
    总之，我们要放的东西是图的某种内部性质和一些辅助的实现（Iterable之类），而不是依赖于某个算法的东西；这能够保持我们代码的层次性

3. 邻接矩阵、邻接表

表示一张图有两种方法：邻接矩阵、邻接表
邻接矩阵：一个V*V（V为节点总数）的矩阵，记录（i, j）边的权值（距离、边长）
对于一个无权图，可以设置为连接是1， 不连接是0
对于一个有权图，可以设置为连接是w(权值)， 不连接是inf
（一条边是不是有向的不重要，如果无方向，看成两条相反的有向边即可）
复杂度O（V*V）

邻接表：
对于每个节点，维护一张表，记录和这个节点相连的节点（和权值）
（表的内部结构是啥无所谓，可以是链表，BST，HashSet之类，按照实践需要定）
复杂度O（V+E）

表示一张图的方法是最重要的部分！这部分的实现效率决定了算法时间复杂度的上界！
在很多问题上，图算法本身的复杂度能够做到O（V+E），E是边数，如果采用邻接矩阵，算法的上界只能是O（VV），一般都是用邻接表，因为现实的图一般都是稀疏的，边数E不会达到VV量级

4. dfs, bfs

我们先来解决s-t问题
s-t Path: Is there a path between vertices s and t?
对任意两点s-t，是否存在一条路径？
更进一步的，如果存在路径，如何找出这条路径？

路径的存在性可以由并查集给出，但是如果采用WQUPC，没有方法给出完整的路径

考虑一步一步找，先找s的相邻点s1，然后再找s1的相邻点s2，……直至所有的点都被遍历；
为了避免s1又找回s（出现环），我们需要对走过的点做标记，所以最后的路径是一颗树！
树有先序、中序、后序和层序遍历，对应在图中就是深度优先搜索（dfs, 先或中或后）或广度优先搜索（bfs, 层序）

实现思路：
dfs: 维护一个栈，挑选一个此刻点的临近点入栈，之后走到那个点上，如果走到死路了，出栈一个点回溯，直到走到目标点；入栈出栈的过程之中更新节点的路径（每个节点记录自己的上一个节点）

bfs：维护一个队列，令此刻点的所有临近点入队，之后出队此刻的点
当队列空或者搜索到目标时，结束搜索
伪代码：

Initialize the fringe (a queue with the starting vertex) and mark that vertex.
Repeat until fringe is empty:
Remove vertex v from the fringe.
For each unmarked neighbor n of v:
Mark n.
Add n to fringe.
Set edgeTo[n] = v.
Set distTo[n] = distTo[v] + 1.

dfs vs bfs，不好说哪个好。
从时间上说：两者的复杂度都是O（V+E），dfs的优点是，如果只需要找一条路径，可能会略微快；bfs的优点是，找到的路径必定是最短的路径

从空间上说：bfs较优，维护一个队列比维护一个可能很长的调用栈轻松（dfs递归容易爆炸）

  • DFS is worse for spindly graphs. Imagine a graph with 10000 nodes all spindly. We’ll end up making 10000 recursive calls, which is bad for space.
  • BFS is worse for “bushy” graphs, because our queue gets used a lot.

5. dijkstra, A*

dijkstra:

如果只是在无权图上找一条路径，dfs和bfs是好的，但如果我们想在有权图上找一条最短路径，无法处理含权值的边的dfs和bfs就不行了

为了解决这个问题，出现了dijkstra算法：

def dijkstras(source):
    PQ.add(source, 0)
    For all other vertices, v, PQ.add(v, infinity)
    while PQ is not empty:
        p = PQ.removeSmallest()
        relax(all edges from p)

def relax(edge p,q):
   if q is visited (i.e., q is not in PQ):
       return

   if distTo[p] + weight(edge) < distTo[q]:
       distTo[q] = distTo[p] + w
       edgeTo[q] = p
       PQ.changePriority(q, distTo[q])

在bfs之中，我们利用了层序遍历的思想，如果根据距离源点的边数多少把节点分成若干层，那么，s到一个在k层的t节点的最短路径一定不可能涉及k+1之后的层

在dijkstra之中，虽然由于权值的影响，不能像bfs那样分层，但依然需要将这个问题分步处理

我们放弃一次性给出从s到某点t的全局最优路径，而是不断的优化（贪心）

对于一个点来说，进入它的所有边都是可能优化其到源点的最小距离的，总共有E条边，所以我们理想的时间界是O（E+V）

dijkstra算法的伪代码如上：

  1. 所有点入队
  2. 找到当前距离s最短的点p
  3. 遍历除p之外所有点，如果w(p,q)+dis[p]<dis[q], 那么更新dis[q]
  4. 从队列之中移除p，如果队列不空，循环23

正确性的证明：
首先，第一步是对的，假设s的相邻点是s1,s2,…，其中w（s, s1）最小，那么s1的最短路径必然是w（s，s1）

第二步，找到此时除s1之外距离s最短者，记为s2，如果存在某条路径，使得此时s到s2不是真正的最短距离，那么
只有一种可能：这条路径的第一步是s->一个除s1,s2的其他点，那么由假设，第一步的路径就已经比此时的最短路径大，矛盾！

所以s2也必然是全局的最短，以此类推

算法复杂性？
如果图是稠密的，那么没啥优化的方法，找最小值就老实扫描一次，是O（V）调用V次是O（V*V），relax对边优化是O（E），总共是
O（V*V+E）=O(V*V)，考虑到稠密图的创建也是这个复杂度，算法复杂度可以接受

如果图是稀疏的，可以建立一个含有各个点的距离s的数据的最小堆，由于图是稀疏的，每个点只连接常数的点，故只需要更新常数次堆，复杂度为O（logV）
总体复杂度就是O（ElogV）

A*：

dijkstra的好处在于，一趟下来，不仅指定s,t的最短路径知道了，我们实际上知道了s与任意一个其他点的最短路径

但很多时候我们不需要知道到其他点的路径，各个方向同权的dijkstra就有点慢了

如果只需要一个点的路径的话，
我们希望设计这样一种算法，使得“试探”路径的时候，更倾向于当前点到终点的方向

显然这样的算法不保证是全局最优的，但我们只需要一个“较优”的答案
In Dijkstra’s, we used bestKnownDistToV as the priority in our algorithm. This time, we’ll use
bestKnownDistToV+estimateFromVToGoal as our heuristic.

在dijkstra之中，我们设法用“已知的最优距离”替代了全局最优距离；在A*之中，我们用“已知最优+从目前点到终点估计距离”替代全局最优

什么是“估计距离”？一种简单泛用的取法是取当前点到目标点的直线距离
“估计距离”让算法带有方向性，这也是A*算法作为启发式算法的特征（有AI的味道）

具体步骤：
对于相邻的所有点，计算从源点到该点的“已知距离”，再加上从该点到目标点的“估计距离”，选取两者和最小的一个点作为实际走的下一个点

正确性：
考虑到终点有一个必经之地p， 现在如果把p的“估计距离”取为无穷，那我们永远到不了终点；
也就是说，A*算法的正确性对“估计距离”的选取有一定的要求

The takeaway here is that heuristics need to be good. There are two definitions required for goodness.

  1. Admissibility. heuristic(v, target) ≤≤ trueDistance(v, target). (Think about the problem above. The true distance from the neighbor of
    C to C wasn’t infinity, it was much, much smaller. But our heuristic
    said it was ∞ , so we broke this rule.)
  2. Consistency. For each neighbor v of w:
    • heuristic(v, target) ≤ dist(v, w) + heuristic(w, target)
    • where dist(v, w) is the weight of the edge from v to w.

6. 最小生成树，割性质

最小生成树（Minimun spanning tree, MST）：
定义：连接一张图所有节点的树叫生成树，最小生成树就是生成树之中各个边的权值加起来最小的树。与一般的树不同的是，最小生成树没有“根”或者“叶”的结构，因为最小生成树是图的一个固有性质，和选择的生成起点无关（这就和dijkstra算法所得到的树不同）

最小生成树常常和以某个点作为起点的dijkstra算法产生的最短路径树相同，但不是始终存在令两者相同的点

割性质：
为了高效的得到最小生成树，需要引入割性质

如图，把一张图分成两个点集AB（AB随意指定），连接A集之中一个点和B集之中一个点的边叫做“割（cut）”

比如图中的（2，7），（1，2），（1，3）

性质：最短的一条割必然是最小生成树的一部分
证明：
反证法：假设不是，存在一棵不经过割w（i,j）的最小生成树，由其经过所有节点的特性知，书中必然存在另一条割连接A，B点集，记为w(p, q)

由于w(i,j)是最短的割，那么w(i,j)<w(p,q)

将w(p,q)+w(i,p)+w(q,j)三段替换成w(p,i)+w(i,j)+w(j,q)依然满足生成树的性质，且更小，矛盾！
证毕

特例：w(i,j)=w(p,q)
同理，w(p,q)+w(i,p)+w(q,j) = w(p,i)+w(i,j)+w(j,q)
若其中一条边在最小生成树之中，必有另一棵最小生成树（按照上式替换），使另一条边在该树之中

而同上，两条边至少有一条在最小生成树之中
所以两条边都在最小生成树之中，可以任取一个（最小生成树可能是同一棵，可能是不同）

垫脚石：单纯运用割性质，我们也能得到一个不好的算法，不断的随机取点集划分，再找到最小割，直到找到了V-1条边

Prims算法:

  1. 取离已经走过的点形成的树最近的一条边
  2. 由割性质（两个点集为走过的和未走的），1取出的边一定是最小生成树的部分
  3. 重复V-1次

算法复杂度：
3是O（V），问题在步骤1时间复杂度是多少
算法1：暴搜，那O（E）不好
算法2：维护一个包含割的优先队列，每次从里面remove一条最短边是O（logE）的，新增边是常数的，因而维护堆也是O（logE）的，
所以总时间复杂度是O（ElogE）= O（ElogV）(E<V*V)的

Kruskal算法：
思想更直观：

  1. 建一个包含所有边的优先队列
  2. 不断的移除最小边，如果形成环，则跳过，否则移除的最小边加入最小生成树（割性质），直至有V-1条边

时间复杂度：
判断是否成环，并查集WQUPC, 稀疏图新节点产生的新边是常数条，对于第一条新边相连的节点，找祖先节点，对于第二条新边相连的节点，找祖先节点，如果两祖先相同则成环，….. 常数两两组合还是常数

建立队列，O（ElogE）= O（ElogV）
判断成环O（Elg*V）(最多E次，每次lg*V)
总共O（Elg*V+ElogE）
如果图API提供了排好序的边，就是O（Elg*V）
在任意有现实意义大小的图的情况下
O（Elg*V）=O（E）！！！

压缩：compress

首先，所有的文件本质上都是字节序列，因而本质上不存在对一个完全随机的文件，可以稳定无损压缩的算法（要保证对能够表示任意N长度字节流，同样也需要N长度）

但对现实世界之中真正存在的文件，都不是真正随机的字节流，比如有的文件存在大量重复的部分（音频，图像之中相同的颜色之类）因而可以构想出类似于“X6”（表示6个X）之类的压缩方法；也有的文件的字节序列有固定的实际意义范围，例如一个纯英文文档不需要使用UNICODE来表示（因而可以在这方面压缩）；在现实情况下，各个符号的出现频率也是不同的，因而如果我们设法使出现频率高的符号用更少的位数表示，也能达到压缩的效果。

讨论对一种进行压缩的通用方法，对文件之中的“符号”进行重新表示

例如，如果有一个文件，只含有“典乐孝绷麻急”，那我们完全可以创造一个表，其中“典”对应字符001，“乐”对应000，“孝”对应010等等，总共我们只需要三位就可以表示这几个字符，而不必用UNICODE（16位每个字符），从而将文件大致压缩到3/16（略大，因为还要包括压缩和解压缩算法、字符对照表）

我们甚至可以做得更好，如果“典”的出现概率为99%，我们可以将“典”设置为0，这样，整个文件的长度将会大大减少。

那么问题来了，在诸如00010101010111这样的流之中，如何确认怎么划分字符呢？（也就是说怎么解码呢？）

方法就是所谓的无前缀编码，任何两个编码不能互相包含，例如不能存在一个编码是0，另一个是01；

例如一种无前缀编码是1，01，001，0001, …假设它们对应ABCD
那么字节流10100100101010001对应
ABCCBBD，没有任何歧义

这种解码的实现方法是伸展树tries，每一次分支左/右代表0/1，而解码的过程就是不断读入字节流，直到伸展树的叶结点， 即为解码了一个字符（无前缀就是在保证所有节点都在叶结点上）

又有了更进一步的问题，这样的编码树有很多种，哪种编码树能够压缩得更小呢？
这个问题复杂在，每个符号（广义的“字符”，但可以引申到图像的像素，音频的单个音，etc）在文件之中出现频率（我们可以扫一遍文件来统计）是不同的，如果让出现的多的符号在编码树浅的地方，那么空间和时间的消耗都会减少。

1. 并查集（Disjoint Set Union）：

先从一个问题引入，我们现在有N个数据，某些数据之间有相连的关系（已知，比如AB相连，BD相连）；现在我们想要判断数据A和数据D是否相连，或者数据C属于哪个集合，怎么做？
（对应的现实情况有很多，比如社交网络等等；并且N经常很大，比如上亿）

最低效的做法肯定是将同类的数据放到一个集合之中，比如
[{A}, {B},{C}]， AC相连就先检索到C，再将C移出集合{C}，放入集合{A}

明显这种做法（ListOfSet）下，对于找数据，添加相连，判断数据是哪个类的都是theta(N)的

改进1（QuickFind）：

集合的操作非常耗时并且不好检索，采用类似数组的结构来对应集合

现有一个初始值为 ABCDEFG的数组，代表1 2 3 4 5 6 7 8 9号（下标）数据

之后如果把 AC相连， 那么我就把下标3的对应值改成A（代表3（对应数据C）现在是1集合（对应A）了， 变成ABADEFG）

这种方法明显地快，但还是O（N）的，比如现在值为
AAAABBBB，让1和5相连，我需要先检索一遍整个数组找到值为B的所有值，再改成A

这个方法的好处在于判断一个数据属于哪个类是O（1）的了，只要看对应值就行

改进2(QuickUnion)：

一种进阶的思路是：

在连接的时候，不再一次改写所有的值（这样的操作很多是无用的，因为要改来改去）

我们只记录和改写一个数据的父节点，形成一个树结构

如图，我要连接5和2，先沿着5的父亲–5的父亲的父亲—这条路找到5的根节点，再找到2的根节点，再将2的根节点的父亲设置为5的根节点

判断一个数据属于哪一类也只要从底层不断沿父亲往上找就行

如果树是平衡的，那么这种方法就是O（logN）(对于连接和判断两种操作)

但这种方法的最坏情况还是theta(N), 设想这样的情况
1 2 3 4 5 5个数据，先让45连，再让34连…最后得到了一个深度 = N的树

改进3：

那我们就想，如何调整树，使它接近于平衡呢？
注意到我们之前合并的时候，连接34实际上无所谓3做父节点还是4做父节点，从深度尽量小的角度来说，我们希望这个选择能让合并的树的深度尽可能小

于是有了Weighted Quick Union(WQU)
我们总是让元素比较多的树的根节点作为父节点，可以证明它是O（log(N)）的，最坏也不会超过logN

(当然还有Heighted Quick Union，让深度较低的树的根作为父节点，但这样也是O（logN）编程却麻烦不少，用得比较少)

改进4：路径压缩（WeightedQuickUnionWithPathCompression）

在查找父节点的过程之中，我们实际上可以把路径之中（A的父亲，祖父, 等等等）全部改成A的最古老的先祖（根节点）的直接的儿子，在下一次查找之中，这一整条路上的所有元素都只需要一次找父亲

这样的复杂度是O（lg* N）的，lg*()是迭代对数，对任意实际情况的N都可以认为它<5, 基本就是O（1）

也就是我们使用O（1）的时间界完成了查找和连接的操作，多美妙！

﻿# 从BST说起
BST（Binary Search Tree, 二叉搜索树）的平均深度是Θ（logN）（一棵随机生成的BST（不管有没有随机插入和删除）的深度和期望高度都是 Θ（logN）的）

但现实之中的树不一定都是随机的，有很多情况（比如带有时间顺序的数据，就不是随机的），我们需要避免Θ（N）的最坏情况，要把树的深度压到O（logN）

二叉树的删除：无儿子、单儿子、双儿子：双儿子通过前驱/后继节点转换为单儿子情况

思路一：

我们给添加节点增加一个规则：不允许“手动加深”树，即不能在新的一层添加新的数据，只允许在已有层添加数据，但我们允许一个节点有最多L个数据（不然设想对一个节点一直加数据，我们的树退化成链表了）

由此诞生了B树，这是一种保证所有的叶节点都是等深的数据结构

B树的检索：基本同二叉搜索树，但在每一个包含L个数据的节点会进行L次比较操作，所以检索操作时O（LlogN） = O(logN)的

B树的插入
先检索到插入的位置插入，这一部分同BST，但可能让插入的节点超过了L个数据的限制，这个时候就需要分裂（让中间的节点而不是其他节点进入上一层比较好）
一棵2-3-4树（表示一个节点可以有2/3/4个子节点）示例：

B树的用处：1.L很小，作为一种简单的平衡搜索 2. L很大，用在数据库、硬盘之类

B树的不足：

  1. 难编程（可以设想L大的时候事情会变得多少复杂，再想想删除有多少种情况）
  2. 很多情况下慢（都是logN 但常数因子大）
  3. 每个节点甚至需要一个链表
    (漂亮的算法不一定是最好的)

拓展：B树的删除，B+树（一颗只在叶子节点存数据，内部结点只做查找索引，并且叶子节点类似链表链接的树）

思路二：

前置：AVL树与旋转

相同节点的BST有许多种形态，而旋转就是从其中一种过渡到另一种的方法

N个节点构成的二叉搜索树的种类是Catalen(N)

对N较大时，任意两个状态之间所需要的旋转数< 2N-6
左旋：将选定节点变成右儿子的左子树
右转：将选定节点变成左儿子的右子树

红黑树：

B树有许多良好的性质，但是它的建立和删除实在是太繁琐了，如果能建立这样的一种树，它是一棵BST，但却和一棵2-3 B树一一对应，形成一个双射，不就综合了两者的优点？

这样的数据结构就是红黑树，也是实际之中运用得最多的树，是java TreeMap() 的底层实现

我们设想将B树含有多个数据的节点可以“暂时分离”，并将分出来的线用红色表示

（如果我们指定分离到更深一层的是df的d或者f，这种分离就是一一对应的，以指定d（左倾红黑树， LLRB，Left-Leaning Red Black BST）为例）

现在有一颗这样搭建好深为N1 = Θ（logN）的B树，将其这样分离，形成的红黑树深度最多是 N2 = N1+1（红线贡献的额外深度）+N1，还是 Θ（logN）的

现在考虑2-3树的插入，插入的分裂操作还没对应红黑树的操作

插入可以分解为：

  1. 插入一个节点
  2. 如果数据数超过L，将中间的数据移动到父节点

直接给出对应规则


那么我们就构建了一棵红黑树！可以总结它的插入规则如下（通过和B树的对应理解！）：

1.（B树）如果有超过数据限制数的节点————>旋转（红黑）
2. （B树）如果要向上合并————> 节点所连的所有线变色（红黑）
3. 如果有连续两条左倾红色连接 ————>旋转（辅助2）

游戏代码设计哲学
2023/10/16

9.14 游戏代码设计哲学：

本质需求：解耦合、降低复杂度

设计模式：

  1. 观察者模式：如果要设计一个“落水”的完整效果，包含声音、特效、物理等，如何解耦模块？
答

将物理模块本身抽出，防止物理模型和UI效果耦合 物理模块处理完落水事件之后产生一个event

让音量模块、特效模块等作为不同的观察者observer，当检测到event产生，执行对应处理代码。

追问：设计应该是同步的还是异步的？什么情况不应该使用这个模式？

  1. 状态模式：善用状态机

    如果有这样一个逻辑：按space跳跃，空中允许二段跳，空中按J下俯冲攻击，俯冲、跳跃、落地均有动画，如何设计？

答
状态机：
       
人物状态：站立、空中1段、空中2段、落地

       
改变状态的act：站立->跳跃->空中1段，空中1/2->俯冲，空中1/2->正常落地，空中1->跳跃->空中2

       
不改变状态的act：其他
如果不定义状态机，你需要记录的“状态"有： 是否空中、是否落地、是否俯冲、跳跃段数......

当状态更多的时候，逻辑会很复杂

假设已经存在act inputhandler(...){}函数， return act

则只需定义这样的函数在主循环中执行


       void updateState(act Act) {
           switch (this->State) {
               case STAND: {
               // 处理站立状态逻辑
                   break;
               }
               case ONAIR_ONE_JUMP: {
               // 处理空中1段跳跃状态逻辑
                   break;
               }
               case ONAIR_TWO_JUMP: {
               // 处理空中2段跳跃状态逻辑
                   break;
               }
               case DIVE: {
               // 处理俯冲状态逻辑
                   break;
               }
           // 其他状态的处理
           }
       }
  1. 数据建模：如果我想要实现一个道具“剑”，但有二十种不同效果的剑，比如冰霜、毒素、眩晕等等，我应该如何设计？
答 不应该设计成一个基类+20个派生类 更好的方法是定义一个data文件（或者微型数据库），里面可以是诸如剑的json格式 

   {
       name: sword,
       prototype: basic_sword,   
       damage: 10,
       type: cut,
       additional_effect: none,
       ...
   }
这样只有当需要时，才查询记录, 并且对所有类似的都可以用一套方法

  1. 如何实现撤销任意步的操作？
答

蠢方法是记录每一帧，但这显然太占用资源；好方法是对每一个操作实现undo接口（如果可能），如果不可能则记录每次的状态，之后重演逻辑、重新渲染
