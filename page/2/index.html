1. 拓扑排序

对于任意的节点i,j 如果存在从i到j的一条边，则j应该排序在i的后面
满足这样性质的排序叫拓扑排序

存在拓扑排序的条件：图是无环的

构建拓扑排序的方法：
对节点引入两个量，入度和出度，代表进入节点和从节点发出的边条数

首先，由于图是无环的，必然可以找到一个入度为0的点（“起点”）
抛去起点和与起点相连的边之后，子图必然有一个新“起点”

不断找这样的“起点”就得到了拓扑排序

算法是O（V+E）的

  1. 计算所有点的入度 O（V+E）
  2. 维护一个队列，包含所有入度为0的点，每次除去一个点和其邻边，并将相邻节点的入度减一（无向图）O（V+E）

python并发
2023/10/16

﻿

并发：多协程、多线程、多进程、多机器

任务：IO密集和cpu密集
IO密集：数据库、文件读写等
cpu密集： 计算密集

多线程：单cpu（前提：python GIL锁），目的是io和cpu分离并行执行，适用于io密集，内存开销比多进程少

多进程：多cpu，目的是多cpu并行计算，适用于cpu密集，内存开销大

线程threading库

t = threading.Thread(target=func, args=(…))
t.start() # 开始线程，执行func
t.join() # 等待线程结束

例如：

threads = []
for _ in range(count):
    thread = threading.Thread(target = f, args=(...))
    threads.append(thread)
for thread in threads:
    thread.start()
for thread in threads:
    thread.join()

如果多个线程之间存在共享资源
需要加锁，
lock = threading.Lock()给需要线程独占的运行函数f套上with lock:

当对一个类的某个方法加锁的时候，需要在__init__上锁self.lock = threading.Lock()
在方法调用之中
with self.lock：
而不是直接在方法调用之中创建lock, 这样无法保证创建lock与线程切换的顺序

生产者-消费者模型：

以爬虫为例：
生产者线程get网络资源，存储到任务序列之中

9.14 游戏代码设计哲学：

本质需求：解耦合、降低复杂度

设计模式：

观察者模式：

如果要设计一个“落水”的完整效果，包含声音、特效、物理等，如何解耦模块？

答

将物理模块本身抽出，防止物理模型和UI效果耦合 物理模块处理完落水事件之后产生一个event

让音量模块、特效模块等作为不同的观察者observer，当检测到event产生，执行对应处理代码。

追问：设计应该是同步的还是异步的？什么情况不应该使用这个模式？

状态模式：善用状态机

如果有这样一个逻辑：按space跳跃，空中允许二段跳，空中按J下俯冲攻击，俯冲、跳跃、落地均有动画，如何设计？

答
状态机：
       
人物状态：站立、空中1段、空中2段、落地

       
改变状态的act：站立->跳跃->空中1段，空中1/2->俯冲，空中1/2->正常落地，空中1->跳跃->空中2

       
不改变状态的act：其他
如果不定义状态机，你需要记录的“状态"有： 是否空中、是否落地、是否俯冲、跳跃段数......

当状态更多的时候，逻辑会很复杂

假设已经存在act inputhandler(...){}函数， return act

则只需定义这样的函数在主循环中执行


       void updateState(act Act) {
           switch (this->State) {
               case STAND: {
               // 处理站立状态逻辑
                   break;
               }
               case ONAIR_ONE_JUMP: {
               // 处理空中1段跳跃状态逻辑
                   break;
               }
               case ONAIR_TWO_JUMP: {
               // 处理空中2段跳跃状态逻辑
                   break;
               }
               case DIVE: {
               // 处理俯冲状态逻辑
                   break;
               }
           // 其他状态的处理
           }
       }

数据建模：

如果我想要实现一个道具“剑”，但有二十种不同效果的剑，比如冰霜、毒素、眩晕等等，我应该如何设计？

答 不应该设计成一个基类+20个派生类 更好的方法是定义一个data文件（或者微型数据库），里面可以是诸如剑的json格式 

   {
       name: sword,
       prototype: basic_sword,   
       damage: 10,
       type: cut,
       additional_effect: none,
       ...
   }
这样只有当需要时，才查询记录, 并且对所有类似的都可以用一套方法

答

蠢方法是记录每一帧，但这显然太占用资源；好方法是对每一个操作实现undo接口（如果可能），如果不可能则记录每次的状态，之后重演逻辑、重新渲染
